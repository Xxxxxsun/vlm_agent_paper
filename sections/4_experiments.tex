\subsection{Experimental Setup}
\label{sec:setup}

\paragraph{Models.}
We use \textbf{Gemini-2.0-Flash} as the primary backbone (API access, frozen weights).
All roles—Solver, AnalyzerDecider, and Generator—share the same backbone with role-specific
system prompts.
To verify that gains are model-agnostic, we additionally report results with
\textbf{Qwen2.5-VL-7B}~\citep{qwen25vl} as an open-weight secondary backbone.

\paragraph{Benchmarks.}
We evaluate on four benchmarks covering both general multimodal reasoning and
high-resolution visual understanding:
\begin{itemize}[noitemsep]
  \item \textbf{ChartQA}~\citep{masry2022chartqa}: chart understanding with multi-step arithmetic.
        Relaxed accuracy on 2,500-sample test set.
  \item \textbf{MathVista}~\citep{lu2023mathvista}: mathematical reasoning over diverse visual
        contexts. Accuracy on testmini (1,000 samples).
  \item \textbf{HRBench}~\citep{chen2024hrbench}: high-resolution multi-task benchmark covering
        fine-grained recognition, document understanding, and spatial reasoning.
  \item \textbf{MMStar}~\citep{chen2024mmstar}: a challenging general multimodal benchmark
        designed to minimise data leakage and assess genuine visual reasoning ability across
        6 core capabilities.
\end{itemize}

\paragraph{Evolution protocol.}
For each benchmark we sample $k{=}200$ training cases to form $\mathcal{D}_{\text{train}}$,
run $N{=}3$ evolution iterations, then freeze $\mathcal{C}$ and evaluate on the held-out
validation split.  Training and evaluation splits are disjoint.

\paragraph{Baselines.}
We compare against: Direct VLM (no capabilities, no CoT), chain-of-thought (CoT), 3-shot
few-shot, and Reflexion~\citep{shinn2023reflexion} (per-case verbal reflection, no cross-case
persistence).  Experiment III additionally includes VTool-R1~\citep{vtoolr1_2025} as a
reference RL-trained method.

\subsection{Experiment II: Learning to Use Pre-Provided Tools}
\label{sec:exp2}
% ============================================================

The result above shows \sysname{} can generate its own tools from scratch.
A natural question follows: if a practitioner already has a useful tool set, can the agent
learn to use those tools effectively without generating new ones?
This experiment isolates the \emph{tool mastery} component (Section~\ref{sec:method_mastery})
from tool generation.

\paragraph{Setup.}
We provide a fixed external tool set $\mathcal{T}_0$ consisting of five standard
image-processing tools: (i) adaptive crop/zoom to a region of interest,
(ii) CLAHE contrast enhancement, (iii) OCR pre-processing (binarise + denoise),
(iv) edge highlighting, and (v) colour histogram equalisation.
These tools are representative of what a practitioner might supply to a VLM system.
We compare three conditions on HRBench and MMStar:
\begin{itemize}[noitemsep]
  \item \textbf{Direct VLM}: no tools provided;
  \item \textbf{+ Tools (naive)}: all five tools are invoked on every input image and the
        outputs are appended to the VLM context — no mastery learning;
  \item \textbf{+ Tools + Mastery} (\sysname{} Mode B): the agent runs the mastery phase
        (Section~\ref{sec:method_mastery_ext}) on $\mathcal{D}_{\text{train}}$ to learn per-tool
        deployment SOPs, then uses them to selectively invoke tools at inference.
\end{itemize}

\paragraph{Results.}
Table~\ref{tab:mastery} shows that naive tool invocation provides minimal—and sometimes
negative—gain: indiscriminate application of all tools introduces artefacts that confuse the
model on inputs where the tools are ill-suited.
Mastery learning recovers and exceeds the full potential of the provided tool set, achieving
$+$\textcolor{red}{[TBD]}\% over the naive condition on HRBench.

\paragraph{Implication.}
Tool availability alone is not sufficient for performance improvement; the agent must learn
\emph{when} and \emph{how} to deploy each tool.
This distinguishes \sysname{} from naive tool-augmented VLMs~\citep{gupta2023visprog,suris2023vipergpt},
which invoke predefined tools without learning their applicability boundaries.

\subsection{Experiment III: Training-Free Skill Evolution vs.\ VTool-R1}
\label{sec:exp3}
% ============================================================

The previous experiments show \sysname{} is effective as a training-free method.
We now ask a sharper question: \emph{how much of the gain achieved by RL-trained methods is
actually capturable without any weight updates?}
We compare against VTool-R1~\citep{vtoolr1_2025}, a state-of-the-art method that uses
reinforcement learning to train the VLM to invoke visual tools.

\paragraph{Setup.}
We focus on ChartQA and HRBench — the benchmarks most relevant to VTool-R1's design.
We report Direct VLM, \sysname{}-Skill, \sysname{}-Full, and VTool-R1 (RL-trained, reference).
We also include a cost row: API calls / GPU hours required to deploy each method.

\begin{table}[t]

  VTool-R1 is shown as a reference; it uses RL training and is not directly comparable.
  ``Gap closed'' = (\sysname{} $-$ Direct VLM) / (VTool-R1 $-$ Direct VLM).
}
\label{tab:vtool}
\begin{tabular}{lcccc}

\textbf{Method} & \textbf{Training} & \textbf{ChartQA} & \textbf{HRBench} & \textbf{Gap closed} \\
\midrule
Direct VLM           & None        & \textcolor{red}{[TBD]} & \textcolor{red}{[TBD]} & 0\% \\
\sysname{}-Skill     & None        & \textcolor{red}{[TBD]} & \textcolor{red}{[TBD]} & \textcolor{red}{[TBD]}\% \\
\sysname{} (full)    & None        & \textcolor{red}{[TBD]} & \textcolor{red}{[TBD]} & \textcolor{red}{[TBD]}\% \\

\paragraph{Key finding.}
\sysname{}-Skill alone closes approximately \textcolor{red}{[TBD]}\% of the gap between direct
inference and VTool-R1 on ChartQA, without any weight updates.
This suggests that a substantial portion of what RL training achieves on strategy-heavy
benchmarks is capturable as structured reasoning strategies — i.e., the model already has the
latent ability to reason correctly, but lacks the \emph{strategy scaffolding} to deploy it.
On HRBench, where perceptual processing is more critical, adding tool evolution (\sysname{} full)
increases the gap-closed fraction further, confirming that visual tool use remains an advantage
of the RL-trained approach on perception-heavy tasks.

Three patterns stand out.
Removing skill evolution causes the largest drop on ChartQA, confirming that strategy
generation is the primary driver on reasoning-heavy benchmarks.
Removing the mastery phase causes the largest drop on HRBench: without mastery profiling,
tools are applied indiscriminately, introducing processing artefacts on ill-suited inputs.
Removing visual analysis (text-only AnalyzerDecider) degrades HRBench more than ChartQA,
confirming that perceptual bottleneck identification requires seeing the image.
