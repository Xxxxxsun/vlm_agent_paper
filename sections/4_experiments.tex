\subsection{Experimental Setup}
\label{sec:setup}

\paragraph{Base VLM.}
We use Gemini-2.0-Flash as the primary backbone (API access, zero weight updates).
All roles—AnalyzerDecider, Generator, and the solving agent $f_{\mathcal{C}}$—run on the same
frozen model, using role-specific system prompts.
We also report results with Qwen2.5-VL-7B~\citep{qwen25vl} as an open-weight secondary backbone
to assess generality across architectures.

\paragraph{Benchmarks.}
We evaluate on four standard VQA benchmarks spanning different visual reasoning demands:
\begin{itemize}[noitemsep]
  \item \textbf{ChartQA}~\citep{masry2022chartqa}: chart understanding requiring data extraction
        and multi-step arithmetic. Evaluation uses relaxed accuracy on the 2,500-sample test set.
  \item \textbf{V*}~\citep{wu2024vstar}: visual search and localisation in high-resolution images.
        Evaluated on the $\sim$500-sample test set using exact-match accuracy.
  \item \textbf{HRBench}~\citep{chen2024hrbench}: multi-task high-resolution benchmark covering
        fine-grained recognition, document understanding, and spatial reasoning.
  \item \textbf{MathVista}~\citep{lu2023mathvista}: mathematical reasoning over diverse visual
        contexts (geometry, charts, tables, scientific figures). Evaluated on testmini (1,000
        samples).
\end{itemize}

\paragraph{Evolution protocol.}
For each benchmark, we sample $k{=}200$ cases from the training split to form
$\mathcal{D}_{\text{train}}$.
We run $N{=}3$ evolution iterations.
The capability set $\mathcal{C}$ is frozen after the final iteration and applied without further
update to $\mathcal{D}_{\text{val}}$ (the held-out validation split).
Training and evaluation splits are disjoint.

\paragraph{Baselines.}
We compare against the following methods:
\begin{itemize}[noitemsep]
  \item \textbf{Direct VLM}: the backbone model prompted with the question and image, no
        capabilities, no chain-of-thought.
  \item \textbf{CoT}: chain-of-thought prompting with a standard ``think step by step'' instruction.
  \item \textbf{3-shot few-shot}: three manually selected in-context examples per benchmark.
  \item \textbf{Reflexion}~\citep{shinn2023reflexion}: per-case verbal reflection on failure,
        without cross-case persistence (re-implemented for visual QA).
  \item \textbf{\sysname{}-Skill}: our system with skill evolution only ($\mathcal{T} = \emptyset$).
  \item \textbf{\sysname{}-Tool}: our system with tool evolution only ($\mathcal{S} = \emptyset$).
  \item \textbf{\sysname{}} (full): dual-modality evolution with skill and tool evolution.
  \item \textbf{VTool-R1}~\citep{vtoolr1_2025} (reference): RL-trained model, reported for
        context; not a direct training-free comparison.
\end{itemize}

\subsection{Cognitive vs.\ Perceptual Bottleneck Analysis}
\label{sec:bottleneck}

A central claim of this paper is that VLM failure modes decompose into two distinct bottleneck
types, and that different capability types are required to address them.
Table~\ref{tab:main} already shows the pattern: on ChartQA and MathVista, the skill-only variant
matches the full system's performance, while on V* and HRBench the full system's tool evolution
provides an additional gain.

Figure~\ref{fig:bottleneck} shows this decomposition as a 2×2 matrix placing each benchmark
at the intersection of skill-effective and tool-effective axes.
ChartQA and MathVista fall in the skill-effective / tool-neutral quadrant: these benchmarks
require structured multi-step reasoning strategies (e.g.\ chart data extraction SOPs, geometry
proof templates) but do not benefit from image manipulation—the baseline model's visual
perception is already adequate once the correct strategy is applied.
V* and HRBench fall in the skill+tool-effective quadrant: these benchmarks require both
strategy improvement and active image manipulation (e.g.\ zooming into specific regions,
contrast enhancement for fine-grained recognition).

Three findings from the ablation merit discussion.
First, removing the mastery phase causes the largest drop on V* and HRBench:
without mastery profiling, tools are invoked indiscriminately, introducing processing artefacts
on inputs they are unsuited to.
Second, removing visual analysis (reducing the AnalyzerDecider to text-only input) causes a
larger drop on V* and HRBench than on ChartQA or MathVista, consistent with the hypothesis that
perceptual bottleneck diagnosis requires seeing the image.
Third, removing failure memory (preventing cross-case skill accumulation) degrades all benchmarks,
confirming that persistent capability accumulation—not just per-case reflection—drives the gains.

\subsection{Evolution Dynamics}
\label{sec:dynamics}

Figure~\ref{fig:convergence} shows accuracy on each benchmark as a function of evolution iteration
(1, 3, 5, 10 iterations on $\mathcal{D}_{\text{train}}$).
Performance improves rapidly in the first 3 iterations and plateaus thereafter,
suggesting that the most generalisable patterns in $\mathcal{D}_{\text{train}}$ are captured early.
This rapid convergence is practically important: most of the gain is realisable with only
$\sim$600 API calls ($k{=}200$ cases, $N{=}3$ iterations, $\leq 3$ LLM calls per case).

Meaningful gains are observed even with $k{=}25$, and performance scales log-linearly up to
