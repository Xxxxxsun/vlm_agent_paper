\subsection{Experimental Setup}
\label{sec:setup}
% ============================================================

\paragraph{Benchmarks.}
We evaluate on four standard visual reasoning benchmarks that collectively span the
cognitive--perceptual bottleneck spectrum.
\textbf{ChartQA}~\citep{masry2022chartqa} requires multi-step numerical reasoning over
chart images (test split: 2{,}500 questions; evaluation metric: relaxed accuracy with a
5\% numeric tolerance).
\textbf{MathVista}~\citep{lu2023mathvista} tests mathematical reasoning grounded in
figures, diagrams, and scientific plots (testmini split: 1{,}000 questions; accuracy).
\textbf{HRBench}~\citep{chen2024hrbench} probes perception on high-resolution
(${\geq}4\mathrm{K}$) images requiring sub-region localisation and fine-grained visual
disambiguation (accuracy).
\textbf{V$^*$}~\citep{wu2024vstar} measures visual search ability: given a query about
an object's property, the agent must locate and attend to the relevant sub-image before
reasoning (${\approx}500$ questions; accuracy).
These four benchmarks were chosen to exercise complementary failure modes: ChartQA and
MathVista are predominantly \emph{cognitive} benchmarks (the raw visual signal is
legible; the bottleneck lies in reasoning strategy), whereas HRBench and V$^*$ require
explicit \emph{perceptual} augmentation (sub-region extraction, resolution enhancement)
that cannot be resolved by strategy alone.

\paragraph{Foundation models.}
Experiment~I evaluates five proprietary and open-weight VLMs at varying scales:
\textbf{GPT-4o}, \textbf{GPT-5.4}, \textbf{Doubao-Seed-2.0}, \textbf{Qwen3.5-397B},
and \textbf{Gemini-3.0-Pro}.
Experiment~III uses the \textbf{Qwen2.5-VL} family (3B, 7B, 32B) to align with the
VTool-R1 evaluation protocol.
All VLM weights are \emph{frozen} throughout; no gradient updates are performed.

\paragraph{Evolution protocol.}
For each benchmark, we sample a training subset of $k{=}200$ cases from the official
training split and run the \sysname{} evolution loop for $N{=}3$ iterations.
At the end of training, the capability set $\mathcal{C}$ is frozen and evaluated on
the held-out validation / test split.
We report the average over three independent random seeds (different $k{=}200$ subsets)
to account for sampling variance; standard deviations are given in
Appendix~\ref{app:impl}.

\paragraph{Baselines.}
For Experiment~I, we compare four evolution configurations that isolate the contribution
of each component:
\textbf{None (Base Agent)} invokes the frozen VLM with no evolved capabilities — the
zero-shot baseline;
\textbf{Skill Only} runs the evolution loop but restricts the Generator to producing
reasoning skills (no tool generation);
\textbf{Tool Only} restricts generation to executable visual tools (no skill generation);
\textbf{Skill $+$ Tool} is the full \sysname{} system (co-evolution of both capability
types).

\paragraph{Evaluation metrics.}
All primary metrics are task-level accuracy on the held-out split.
Experiment~II additionally reports four step-level metrics on the GTA benchmark:
\textbf{ToolAcc} (fraction of steps with correct tool selection),
\textbf{ArgAcc} (correct argument specification given the correct tool),
\textbf{InstAcc} (overall instruction-following fidelity), and
\textbf{AnsAcc} (final answer accuracy).
Experiment~III follows the VTool-R1 evaluation protocol~\citep{vtoolr1_2025} on Chart
and Table reasoning splits.

% ============================================================
\subsection{Experiment~I: Autonomous Evolution from Scratch}
\label{sec:exp1}
% ============================================================

\paragraph{Motivation.}
The central claim of \sysname{} is that a VLM agent can bootstrap useful capabilities
from failure on a small, unlabelled training subset, without any weight updates.
Experiment~I tests this claim in the most challenging setting: the capability library
is initialised to empty ($\mathcal{S}{=}\emptyset$, $\mathcal{T}{=}\emptyset$), and the
system must generate, validate, and accumulate all capabilities autonomously.
By evaluating five foundation models ranging from 3B to frontier scale, we assess
whether the gains are model-specific or a general property of the failure-driven
evolution paradigm.

\paragraph{Multi-model comparison.}
Table~\ref{tab:evolution_full} reports accuracy for all model--strategy combinations.
Several consistent patterns emerge.
First, joint Skill~$+$~Tool co-evolution (bottom row of each model block) achieves
the highest accuracy across all benchmarks and all models, confirming that the two
capability types are complementary rather than redundant.
Second, Skill~Only evolution delivers the majority of the gain on ChartQA and
MathVista: for both benchmarks, the delta from None to Skill~Only is substantially
larger than the additional delta from Skill~Only to Skill~$+$~Tool.
This pattern is consistent across all five models and supports the hypothesis that
ChartQA and MathVista are primarily \emph{strategy-bottlenecked} — the VLM already
has the visual acuity to perceive the relevant information, but lacks a structured
reasoning procedure for extracting and operating on it.
Third, the relative ordering flips on HRBench and V$^*$: Tool~Only provides
non-trivial gains on both benchmarks, and Skill~Only alone falls measurably short
of the Skill~$+$~Tool ceiling.
This asymmetry indicates a \emph{perceptual bottleneck}: the raw image resolution or
sub-region localisation difficulty exceeds what the frozen VLM can handle without
external preprocessing, regardless of how refined its reasoning strategy is.
The cognitive--perceptual decomposition is explored in detail in
Section~\ref{sec:ablation}.

The absolute magnitude of improvement varies with base model capability.
Weaker models (e.g.\ lower-scale open-weight variants) benefit more from both skill and
tool evolution, likely because they have more headroom for improvement and fewer implicit
reasoning heuristics that the skill library must override.
Frontier models (GPT-5.4, Gemini-3.0-Pro) show smaller but still consistent absolute
gains, suggesting that even highly capable VLMs leave performance on the table without
task-specific strategic knowledge.

\paragraph{Cumulative streaming setting.}
A practically important question is whether \sysname{} can adapt \emph{continuously} as
the task distribution shifts, rather than evolving from scratch for each benchmark
independently.
To test this, Table~\ref{tab:streaming_accumulation} evaluates five strategies under a
\emph{streaming accumulation} protocol: benchmarks are introduced sequentially (HRBench
$\rightarrow$ MathVista $\rightarrow$ V$^*$ $\rightarrow$ ChartQA), and the capability
library is never reset between benchmarks.
Each column reports accuracy on all tasks seen up to that point.

The \emph{Dynamic Skill--Tool Co-Evolution} row (our full system) maintains the highest
\textbf{Final Avg.} score and shows no catastrophic forgetting on earlier benchmarks as
new ones are introduced.
By contrast, \emph{Naive Skill+Tool Accumulation} — which adds capabilities without
any conflict-resolution or mastery re-profiling — shows degraded accuracy on earlier
benchmarks after later benchmarks are introduced.
This suggests that unconstrained accumulation can introduce \emph{capability
interference}: a skill generalised from one task family may trigger incorrectly on a
different one.
The co-evolution framework mitigates this by continuously revalidating existing
capabilities against the growing training distribution (regression test in
Section~\ref{sec:method_tool}), ensuring that newly accumulated knowledge does not erode
prior competencies.
The skill-only and tool-only ablations behave analogously to Table~\ref{tab:evolution_full}:
both fall short of the co-evolution baseline on perceptual benchmarks, confirming that
neither capability type alone is sufficient for a diverse, growing task distribution.

\subsection{Experiment~II: Learning to Use Pre-Provided Tools}
\label{sec:exp2}
% ============================================================

\paragraph{Motivation.}
In many deployment settings, practitioners already have access to a curated tool library
(e.g.\ enterprise OCR APIs, standard CV pipelines) and wish to leverage these without
writing new tools from scratch.
A natural question is: given a high-quality external tool set $\mathcal{T}_0$, does
a VLM agent benefit more from (a) invoking all tools unconditionally, or (b) learning
\emph{when} to invoke each tool through mastery profiling?
Experiment~II addresses this question using the GTA benchmark, which provides a
structured evaluation of step-level tool-use fidelity in addition to final answer
accuracy.

The key finding this experiment targets is the \emph{mastery gap}: naive tool invocation
is insufficient, and can in fact degrade accuracy on inputs where the tool is
inapplicable.
\sysname{} Mode~B (Section~\ref{sec:method_mastery_ext}) resolves this by running the
mastery profiling phase on a small training set, learning per-tool applicability SOPs
that guide conditional invocation at test time.

\paragraph{Results on GTA.}
Table~\ref{tab:gta_detailed} reports step-level and task-level metrics for four methods
across two foundation models.
\emph{Base Agent} invokes no tools.
\emph{Prompting Only} uses chain-of-thought prompting to encourage structured
reasoning but provides no executable tools.
\emph{Tool Usage Tuning} applies all tools unconditionally, corresponding to naive
tool invocation without any deployment strategy.
\emph{Skill Evolution (Ours)} applies \sysname{} Mode~B: skill evolution proceeds
normally, and the mastery phase learns per-tool SOPs from the training set.

The step-level metrics reveal a pattern that final-answer accuracy alone would mask.
\emph{Tool Usage Tuning} achieves high \textbf{ToolAcc} — the correct tool is
frequently selected — but incurs a substantial drop in \textbf{ArgAcc} and
\textbf{InstAcc} relative to \sysname{}.
This indicates that the naive strategy knows \emph{what} tools exist but not \emph{how}
to parameterise them for specific inputs, nor when \emph{not} to use them.
Over-invocation introduces processing artefacts on unsuitable inputs, which propagates
to lower \textbf{AnsAcc} despite the high raw tool selection rate.

\sysname{}'s mastery SOPs correct this by encoding both positive patterns (input types
where the tool reliably improves the answer) and negative patterns (input types where
the tool should be skipped).
The resulting \textbf{AnsAcc} improvement over \emph{Tool Usage Tuning} is consistent
across both GPT-4o and GPT-5.4, and the gap is larger on GPT-4o — a model with a
somewhat lower intrinsic visual acuity — than on GPT-5.4, suggesting that mastery
learning is more valuable when the base model is more susceptible to tool-induced
artefacts.

\subsection{Experiment~III: Closing the Gap to VTool-R1}
\label{sec:exp3}
% ============================================================

\paragraph{Motivation.}
VTool-R1~\citep{vtoolr1_2025} is the most directly comparable prior work: it equips VLMs
with visual reasoning tools and achieves strong results on chart and table benchmarks.
Crucially, VTool-R1 achieves these results via GRPO-based reinforcement
learning~\citep{vtoolr1_2025}, requiring a significant compute budget (multiple GPU
days of RL training) and model weight updates.
Experiment~III asks: how much of VTool-R1's performance gain over a pure-run baseline
can be recovered by training-free skill evolution alone, at zero training cost?

This comparison is deliberately asymmetric in cost: VTool-R1 consumes training
trajectories and performs gradient-based optimisation, whereas \sysname{} uses only
in-context API calls with no weight updates.
The \textbf{\#Train Traj.}\ column in Table~\ref{tab:vtoolr1_aligned_full} makes this
resource contrast explicit.
We evaluate on the same Qwen2.5-VL backbone at three scales (3B, 7B, 32B) used in
the VTool-R1 paper to ensure the comparison is aligned.

\paragraph{Results.}
Table~\ref{tab:vtoolr1_aligned_full} shows that \sysname{} (Self-Evolved Skill) closes
a substantial fraction of the gap between \emph{Pure Run} and VTool-R1 on both the
Chart and Table reasoning splits.
The absolute gap to VTool-R1 narrows as model scale increases: at 32B,
\sysname{} is within a few percentage points of the RL-trained baseline on the Chart
split, whereas at 3B the gap is larger.
This scale interaction is consistent with the hypothesis that skill evolution is most
effective when the underlying VLM already has sufficient representational capacity to
execute a sophisticated reasoning SOP; smaller models may lack the instruction-following
fidelity needed to reliably apply multi-step skills retrieved from the library.

The comparison also reveals what RL training \emph{actually buys} on these benchmarks.
Because skill evolution alone — which introduces no new visual processing capability,
only refined reasoning strategies — accounts for a large fraction of VTool-R1's gain,
the remaining gap is likely attributable to: (a)~the stochastic policy optimisation
that RL provides (improving robustness of tool invocation), and (b)~weight-level
knowledge transfer from the RL training signal that cannot be replicated in-context.
The relative magnitude of (a) vs.\ (b) is an open question, but the size of the
training-free component suggests that \emph{strategy}, not raw perceptual capability,
is the primary bottleneck on these benchmarks — consistent with the cognitive--perceptual
decomposition from Experiment~I.

\paragraph{Cost comparison.}
\sysname{}'s evolution run on a single benchmark requires approximately 3M input tokens
(200 training cases $\times$ 3 iterations $\times$ $\sim$5K tokens per case including
diagnosis and generation calls), amounting to a few USD at current API pricing.
By contrast, RL training for VTool-R1 requires multiple GPU-days of compute.
The \textbf{\#Train Traj.}\ column confirms that \sysname{} processes dramatically
fewer training signals than VTool-R1 while recovering the majority of the performance
gain, making it practical for low-resource and privacy-sensitive deployment scenarios
where RL training is infeasible.

\subsection{Ablation Studies}
\label{sec:ablation}
% ============================================================

\paragraph{Component ablations.}
To isolate the contribution of each design decision, Table~\ref{tab:ablation_main}
evaluates variants of \sysname{} by removing one component at a time.
We conduct ablations on ChartQA (a representative cognitive benchmark) and HRBench
(a representative perceptual benchmark), and report results for GPT-4o.

\textbf{w/o Skill Evolution} removes the skill generation pathway, restricting the agent
to tool evolution only.
On ChartQA, this ablation incurs the largest accuracy drop, confirming that ChartQA
is primarily a cognitive benchmark where strategy, not perception, is the binding
constraint.
On HRBench, the drop is smaller, consistent with the finding that HRBench requires
perceptual tools that skill evolution alone cannot provide.

\textbf{w/o Tool Evolution} is the symmetric ablation.
Its impact is negligible on ChartQA (consistent with the Skill~Only~$\approx$~Skill$+$Tool
observation from Table~\ref{tab:evolution_full}) but substantial on HRBench, where
executable image processing is the primary bottleneck.

\textbf{w/o Mastery Phase} disables the mastery profiling step: generated tools are
deployed unconditionally on all inputs rather than selectively according to their
mastery SOP.
The accuracy drop is moderate on ChartQA but pronounced on HRBench.
Qualitative inspection reveals that unconditional tool application introduces processing
artefacts (e.g.\ over-cropped sub-regions, over-enhanced contrast) on inputs that would
be handled correctly by the base agent, degrading cases the base agent already solves.
This confirms the hypothesis from Experiment~II: tool provision without mastery guidance
is actively harmful on a non-trivial fraction of inputs.

\textbf{w/o Visual Diagnosis} replaces the multimodal AnalyzerDecider with a text-only
variant that receives only the question, answer, and reasoning trace — not the original
image or tool outputs.
Accuracy drops on both benchmarks, but the drop is larger on HRBench than on ChartQA.
On HRBench, the text-only diagnostician systematically misattributes perceptual failures
(which are visible in the image) as cognitive failures (which would be invisible to a
text-only observer), leading it to generate reasoning skills for cases that actually
require visual tools.
This mis-routing wastes capability budget and accumulates the wrong type of knowledge
in the library.
The result directly supports Contribution~(4) from the Introduction: visual
failure analysis is a necessary, not merely convenient, component of accurate bottleneck
diagnosis.

\paragraph{Sensitivity analysis.}
Appendix~\ref{app:results} reports accuracy as a function of (a)~number of evolution
iterations $N \in \{1, 3, 5, 10\}$ and (b)~training set size $k \in \{10, 25, 50, 100,
200\}$.
Both experiments are conducted on ChartQA.
Key observations: gains plateau around $N{=}5$, and the system achieves ${\geq}80\%$ of
its final improvement with as few as $k{=}50$ training cases, indicating that the method
is effective even in extremely low-resource settings.
